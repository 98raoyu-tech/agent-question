### 标准答案

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将外部知识检索与大语言模型生成能力相结合的技术架构。其核心思想是：在 LLM 生成回答之前，先从外部知识库中检索与用户问题相关的文档片段，然后将这些片段作为上下文（Context）注入到 Prompt 中，引导 LLM 基于真实数据生成更准确、更可靠的回答。

RAG 解决了 LLM 的三大核心痛点：
1. **知识过时**：LLM 的训练数据有截止日期，无法获取最新信息。
2. **幻觉问题**：LLM 在缺乏知识时会"编造"看似合理但错误的答案。
3. **领域缺失**：LLM 缺乏企业私有知识（如内部文档、产品手册）。

**完整工作流程分为五个阶段：**

**阶段一：离线索引构建（Indexing）**
- 文档加载（Document Loading）：从 PDF、Word、网页、数据库等数据源加载原始文档。
- 文档分块（Chunking）：将长文档切分为适当大小的文本块（Chunks）。
- 向量化（Embedding）：使用 Embedding 模型将每个文本块转换为高维向量。
- 存储入库（Storing）：将向量及其元数据存入向量数据库，构建索引。

**阶段二：在线查询处理（Query Processing）**
- 用户输入自然语言问题。
- 对用户问题进行查询重写（Query Rewrite），提升检索效果。
- 使用相同的 Embedding 模型将问题转换为查询向量。

**阶段三：检索（Retrieval）**
- 在向量数据库中执行相似度搜索，召回 Top-K 个最相关的文本块。
- 可选：结合 BM25 关键词检索进行混合检索（Hybrid Search）。

**阶段四：增强（Augmentation）**
- 对检索结果进行重排序（ReRank），提升相关性。
- 将高相关性的文本块拼接到 Prompt 模板中。
- 构造完整的上下文信息。

**阶段五：生成（Generation）**
- 将增强后的 Prompt 发送给 LLM。
- LLM 基于上下文生成最终回答。
- 返回结果给用户。

### 深度解析

RAG 的本质是一种 **"先检索、后生成"** 的范式。它不要求 LLM 记住所有知识，而是让 LLM 具备"查阅资料"的能力。这与人类的"开卷考试"模式高度类似——你不需要背诵整本书，但你需要知道去哪里找答案，并且能够理解和整合找到的信息。

从技术架构看，RAG 的关键在于 **三个匹配**：
1. **Embedding 匹配**：查询向量与文档向量必须使用同一模型编码。
2. **语义匹配**：检索到的内容必须与用户意图高度相关。
3. **上下文匹配**：注入 Prompt 的信息量必须在 LLM 上下文窗口范围内。

RAG 的核心优势在于其 **动态性**——知识库可以随时更新，无需重新训练模型。这使得 RAG 特别适合知识频繁变化的企业场景。

从系统复杂度看，RAG 涉及的技术栈包括：文档解析、文本分块、Embedding 模型、向量数据库、检索算法、重排序模型、Prompt 工程和 LLM 调用。每个环节的优化都会直接影响最终效果。

## Q02 Hybrid Search的原理是什么？

### 🎤 30秒回答版

Hybrid Search结合向量检索和关键词检索，兼顾语义理解和精确匹配。

### 🎤 1分钟回答版

Hybrid Search是混合检索策略：先用BM25做关键词检索，再用向量检索做语义匹配，最后用RRF融合结果。解决单一检索方式的局限性，提升召回率和准确率。

### 🎤 3分钟深度回答版

**原理**：向量检索擅长语义理解，BM25擅长精确匹配，Hybrid Search结合两者优势。

**实现**：分别执行两种检索，用Reciprocal Rank Fusion融合结果。

**优势**：既保证语义相关性，又保证关键词精确匹配。

**调优**：调整两种检索的权重，通过A/B测试找到最优配置。

**适用场景**：企业知识库、电商搜索等需要兼顾精确和模糊匹配的场景。

### 标准答案

**Hybrid Search（混合检索）** 是将向量检索（Semantic Search）与关键词检索（Keyword Search）相结合的检索策略。其核心思想是：通过多种检索方式获取候选文档，然后通过融合策略合并结果，以兼顾语义理解和精确匹配的优势。

**常用的混合检索策略：**

**1. 并行检索 + 结果融合**
- **向量检索**：使用Embedding模型将查询和文档向量化，通过余弦相似度匹配语义相关文档。
- **关键词检索**：使用BM25或TF-IDF算法匹配关键词。
- **融合方式**：
  - **RRF（Reciprocal Rank Fusion）**：对每个文档的排名取倒数求和，再排序。
  - **线性融合**：对两种检索的分数加权求和。
  - **投票机制**：根据多个检索结果的重叠度确定最终排名。

**2. 分层检索**
- **第一层**：使用关键词检索快速筛选候选集。
- **第二层**：使用向量检索在候选集中精排。

**3. 交叉检索**
- 先使用向量检索召回候选，再用关键词检索在候选集中精排。
- 或反之。

**优势：**
- **召回率高**：结合两种检索的优势，覆盖更多相关文档。
- **鲁棒性强**：对不同类型的查询都有较好的表现。
- **灵活性高**：可以根据业务需求调整权重。

**挑战：**
- **性能开销**：需要执行多次检索，增加响应时间。
- **参数调优**：需要找到合适的融合权重。

### 深度解析

Hybrid Search 的本质是 **"博采众长"**——结合不同检索算法的优势，提升整体检索效果。

**为什么需要 Hybrid Search？**
- **向量检索的局限性**：对关键词精确匹配能力较弱，例如"iPhone 15"和"iPhone15"可能被认为不相关。
- **关键词检索的局限性**：缺乏语义理解能力，例如"如何制作蛋糕"和"烘焙教程"可能被认为不相关。

**RRF 融合公式：**
```
score(d) = sum(1 / (k + rank_i(d))) for all retrieval methods i
```
其中 `k` 是常数（通常设为60），`rank_i(d)` 是文档 `d` 在第 `i` 种检索方法中的排名。

**最佳实践：**
1. 使用 Elasticsearch 或 Solr 处理关键词检索。
2. 使用 Pinecone 或 Milvus 处理向量检索。
3. 在应用层实现结果融合。
4. 通过 A/B 测试确定最佳权重配置。

## Q03 ReRank的作用是什么？如何实现？

### 🎤 30秒回答版

ReRank对检索结果重新排序，提升相关性。

### 🎤 1分钟回答版

ReRank使用Cross-Encoder模型对初步检索结果重新评分排序。先召回较多候选，再精排提升相关性。常用模型如BERT-based Cross-Encoder。

**模型选择**：根据性能要求选择不同大小的模型。

**优化**：缓存热门查询的ReRank结果，减少计算量。

**效果**：ReRank可提升5-10%的检索精度。

### 标准答案

**ReRank（重排序）** 是在初步检索结果的基础上，使用更精确的模型对候选文档进行重新评分和排序的过程。其核心目标是：在保持高召回率的同时，显著提升结果的相关性和排序质量。

**常用的ReRank模型：**

**1. Cross-Encoder**
- **原理**：将查询和文档拼接后输入Transformer模型，直接输出相关性分数。
- **优势**：建模能力强，排序精度高。
- **劣势**：计算成本高，无法批量处理。

**2. ColBERT**
- **原理**：使用交互式注意力机制，计算查询token与文档token的细粒度匹配。
- **优势**：兼顾精度和效率。
- **劣势**：实现复杂度较高。

**3. Sentence-BERT Cross-Encoder**
- **原理**：基于Sentence-BERT的Cross-Encoder变体。
- **优势**：预训练模型丰富，易于使用。

**ReRank流程：**

```
1. 初步检索：使用向量检索或混合检索召回Top-N候选文档（通常N=100）
2. ReRank：使用Cross-Encoder对每个候选文档评分
3. 排序：根据评分重新排序
4. 返回：取Top-K（通常K=10）作为最终结果
```

**性能优化策略：**
- **缓存机制**：缓存热门查询的ReRank结果。
- **模型量化**：使用INT8量化减少计算量。
- **异步处理**：将ReRank放在异步队列中处理。
- **分级策略**：对简单查询跳过ReRank。

### 深度解析

ReRank 是检索系统中的 **"精排"** 环节，通常与"召回"环节配合使用。

**召回 vs 精排：**
- **召回**：从海量文档中快速筛选出候选集，追求高召回率。
- **精排**：对候选集进行精细排序，追求高精确率。

**Cross-Encoder vs Bi-Encoder：**
- **Bi-Encoder**：分别编码查询和文档，计算向量相似度。优点是速度快，可预计算；缺点是精度较低。
- **Cross-Encoder**：同时编码查询和文档，直接预测相关性。优点是精度高；缺点是速度慢，无法预计算。

**为什么需要ReRank？**
- 召回阶段为了保证召回率，通常会返回较多候选文档。
- 这些候选文档的排序质量可能不高。
- ReRank可以显著提升排序质量，特别是对于复杂查询。

### 标准答案

**Query Rewrite（查询改写）** 是通过修改用户查询来提升检索效果的技术。其核心目标是：将用户的自然语言查询转换为更适合检索系统处理的形式，从而提高召回率和准确率。

**常见的Query Rewrite策略：**

**1. 查询扩展（Query Expansion）**
- **同义词替换**：将查询中的词替换为同义词。
- **上位词扩展**：添加更通用的词汇。
- **下位词扩展**：添加更具体的词汇。
- **相关词扩展**：基于语料库统计添加相关词汇。

**2. 查询改写（Query Paraphrasing）**
- **句式转换**：将疑问句转换为陈述句。
- **主动被动转换**：调整语态。
- **简化表达**：去除冗余词汇。
- **复杂查询拆分**：将复杂查询拆分为多个简单查询。

**3. 多查询生成（Multi-Query Generation）**
- 使用LLM生成多个与原查询语义相关的查询变体。
- 将多个查询的检索结果合并。

**4. 拼写纠错（Spell Correction）**
- 检测并纠正拼写错误。
- 常用方法：编辑距离、语言模型。

**5. 实体链接（Entity Linking）**
- 将查询中的实体链接到知识库中的实体。
- 例如：将"苹果"链接到"苹果公司"或"苹果水果"。

**实现方式：**
- **规则-based**：基于词典和规则进行改写。
- **机器学习**：训练模型预测最佳改写方式。
- **LLM-based**：使用大语言模型直接生成改写后的查询。

### 深度解析

Query Rewrite 是检索系统中的 **"预处理"** 环节，直接影响后续检索的效果。

**为什么需要Query Rewrite？**
- 用户查询可能不规范、不完整或有歧义。
- 不同用户的表达方式差异很大。
- 查询与文档的表达方式可能不一致。

**LLM-based Query Rewrite的优势：**
- 能够理解上下文和语义。
- 能够生成多样化的改写结果。
- 无需手动编写规则。

**评估指标：**
- **召回率变化**：改写后召回率是否提升。
- **精确率变化**：改写后精确率是否提升。
- **人工评估**：邀请用户评估改写质量。

**最佳实践：**
1. 使用LLM生成多个改写版本。
2. 对每个版本进行检索。
3. 合并去重结果。
4. 选择最优的改写策略。

### 标准答案

**Multi-Query Retrieval（多查询检索）** 是一种通过生成多个查询变体来提升检索召回率的技术。其核心思想是：基于用户的原始查询，生成多个语义相关但表达方式不同的查询，分别进行检索，然后合并结果。

**工作流程：**

```
1. 查询生成：使用LLM基于原始查询生成N个（通常3-5个）查询变体
2. 并行检索：对每个查询变体独立执行检索
3. 结果合并：合并所有检索结果，去重并重新排序
4. 返回结果：返回最终的合并结果
```

**查询生成策略：**
- **同义词替换**：替换查询中的关键词。
- **句式转换**：改变查询的表达方式。
- **角度变换**：从不同角度重新表述查询。
- **扩展/收缩**：添加或删除查询中的限定词。

**结果合并策略：**
- **简单合并**：直接合并所有结果，去重后按某种规则排序。
- **投票机制**：根据文档在多个查询结果中的出现次数排序。
- **分数融合**：融合多个查询的检索分数。

**优势：**
- **召回率提升**：通过多个查询覆盖更多相关文档。
- **鲁棒性增强**：对查询表达方式不敏感。
- **无需人工干预**：自动生成查询变体。

**挑战：**
- **性能开销**：需要执行多次检索。
- **结果冗余**：可能返回大量重复或低质量结果。
- **查询质量**：生成的查询变体质量参差不齐。

### 深度解析

Multi-Query Retrieval 是一种 **"广撒网"** 策略——通过多个查询覆盖更多潜在相关的文档。

**为什么需要Multi-Query？**
- 用户查询可能无法准确表达真实意图。
- 不同的查询表达方式可能匹配到不同的文档。
- 单一查询可能遗漏重要的相关文档。

**查询数量的选择：**
- 太少：无法有效提升召回率。
- 太多：增加性能开销，可能引入噪音。
- 经验值：3-5个查询变体通常是比较好的平衡点。

**与Query Rewrite的区别：**
- **Query Rewrite**：生成一个"最优"的改写查询。
- **Multi-Query**：生成多个查询变体，并行检索后合并。

**最佳实践：**
1. 使用LLM生成3-5个查询变体。
2. 对每个变体进行向量检索。
3. 使用RRF融合多个检索结果。
4. 返回Top-K结果。

### 评测体系
| 指标 | 说明 | 计算方式 |
|------|------|---------|
| Recall@k | 检索召回率 | 相关文档被召回的比例 |
| Precision@k | 检索精确率 | 召回文档中相关的比例 |
| MRR | 平均倒数排名 | 衡量检索结果排序质量 |
| NDCG | 归一化折损累计增益 | 考虑位置权重的排序质量 |
| Faithfulness | 忠实度 | 回答与源文档的一致性 |

### 成本优化
- **语义缓存**：缓存相似查询的结果，避免重复计算
- **模型路由**：简单问题使用轻量模型，复杂问题使用大模型
- **量化技术**：使用INT8/FP8量化减少资源消耗
- **批处理**：批量处理相似请求，提升吞吐量

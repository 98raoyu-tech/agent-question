### 🎤 1分钟回答版

### 需求分析

企业知识库系统是RAG（Retrieval-Augmented Generation）技术最典型的企业级应用场景。其核心目标是将企业内部海量非结构化文档转化为可智能问答的知识服务。

**功能需求：**

- **多格式文档支持**：支持PDF、Word、Excel、PPT、Markdown、HTML网页、Confluence/Wiki等企业常用文档格式。每种格式需要专门的解析器，例如PDF需要处理扫描件OCR、表格识别、公式解析；Excel需要处理多Sheet结构化数据转自然语言描述。
- **智能问答能力**：支持单轮精确查询和多轮上下文对话。多轮对话需要维护对话历史窗口，支持指代消解（如"它的价格是多少"中的"它"需要关联上文提到的产品）。
- **权限控制**：支持部门级、角色级、文档级的细粒度权限。不同部门的员工只能检索到自己有权限访问的文档，权限变更需要实时生效。
- **规模要求**：日均10万次查询请求，文档总量100万+，单文档最大支持200MB，峰值QPS 500。

**非功能需求：**

- **延迟要求**：P99端到端响应时间<3秒（含LLM生成时间），检索阶段P99<500ms。
- **可用性**：SLA 99.9%，年停机时间<8.76小时。
- **准确性**：检索召回率>90%，答案准确率>85%（人工评估）。
- **安全性**：数据传输全程加密，文档内容不可泄露到外部LLM服务（私有化部署或数据脱敏）。

### 核心架构

整体架构分为三大模块：**离线索引Pipeline**、**在线检索Pipeline**、**管理后台**。

### 离线索引Pipeline

```
文档上传 → 格式检测 → 文档解析 → 文本清洗 → 智能分块 → Embedding向量化 → 写入Milvus
    ↓                                                              ↓
文档元数据 → PostgreSQL                                        向量+原文 → Milvus
```

**关键设计决策：**

1. **文档解析层**：采用插件化架构，每种文档格式对应一个Parser插件。核心解析引擎选用Unstructured.io，它统一了PDF、Word、HTML等格式的解析接口。对于扫描件PDF，集成PaddleOCR进行OCR识别。解析后的中间结果（结构化JSON）缓存到对象存储（MinIO/S3），避免重复解析。

2. **智能分块策略**：采用混合分块策略。首先按文档结构分块（章节、段落），然后对超长段落进行二次滑动窗口分块（chunk_size=512 tokens，overlap=64 tokens）。对于表格数据，整个表格作为一个chunk并附加自然语言描述。对于代码片段，按函数/类为单位分块。每个chunk附加元信息（文档ID、章节路径、页码）。

3. **向量化Pipeline**：使用消息队列（Kafka）解耦解析和向量化。文档解析完成后发送消息到Kafka Topic，消费者批量拉取chunk进行Embedding（batch_size=64），利用GPU集群并行处理。单次Embedding延迟<10ms/batch。

### 在线检索Pipeline

```
用户查询 → 查询重写 → 向量化 → Hybrid检索 → ReRank → Prompt构建 → LLM生成 → 结果返回
              ↓                              ↓
         历史对话                         Milvus + Elasticsearch
```

**关键设计决策：**

1. **查询预处理**：使用LLM进行查询重写，生成2-3个查询变体进行多查询检索。同时进行实体识别和意图分类，为检索提供额外语义信息。

2. **混合检索策略**：
   - 向量检索：使用Milvus进行语义相似度搜索，召回Top-100候选
   - 关键词检索：使用Elasticsearch进行BM25检索，召回Top-100候选
   - 结果融合：使用RRF（Reciprocal Rank Fusion）算法融合两个检索结果

3. **重排序**：使用Cross-Encoder模型对融合后的候选进行精排，取Top-10作为上下文注入Prompt。

## 📊 系统设计专项补充

### 容量估算方法论

1. **需求分析**：明确业务指标（QPS、数据量、延迟要求）
2. **单节点能力评估**：测试单服务器/单GPU处理能力
3. **集群规模计算**：总量 / 单节点能力 × 冗余系数(1.5)
4. **存储估算**：数据量 × 副本数 × 增长系数(1.3)

### QPS估算公式

```
服务器数量 = 峰值QPS / 单服务器QPS × 冗余系数(1.5)
GPU数量 = 峰值QPS / 单GPU处理能力 × 冗余系数(1.5)
```

### Token成本计算公式

```
月Token消耗 = 日均查询量 × 平均Token数 × 30
月成本 = 月Token消耗 / 1000 × 单价($/1K Token)
```

### 缓存命中率优化

- 设置合理的缓存过期时间（热门数据1小时，普通数据10分钟）
- 使用多级缓存（内存+Redis）
- 实现语义缓存（相似查询复用结果）
- 定期清理冷数据

### 存储规模估算

```
向量存储 = Chunk数量 × 向量维度 × 4字节
文档存储 = 文档数量 × 平均文档大小 × 副本数
日志存储 = 日均日志量 × 保留天数
```
